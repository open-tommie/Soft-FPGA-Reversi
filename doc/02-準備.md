# 02-準備

開発を始める前に必要なホスト側セットアップ。Phase 1（Hello UF2 ビルド）と
Phase 2（Pico Debug Probe 経由の SWD 書き込み）と Phase 3（Verilator 出力の
firmware リンク）を WSL2 から完結させる。

## 前提環境

| 項目 | 値 |
| --- | --- |
| ホスト OS | Windows 10 / 11 |
| 開発側 | WSL2 (Ubuntu 24.04 推奨) |
| Docker | Docker Desktop または WSL2 内 docker (engine 24+) |
| Docker Compose | v2 系 (`docker compose` サブコマンド) |
| ハードウェア | Raspberry Pi Pico 2 (RP2350)。任意で Debug Probe |

WSL2 内で `docker --version && docker compose version` が通れば OK。

## ツールチェイン (Docker イメージ内、固定)

[`docker/Dockerfile`](../docker/Dockerfile) でピン留めしている。**変更時は組合せ全体を再検証**すること。

| ツール | バージョン | 備考 |
| --- | --- | --- |
| Pico SDK | 2.2.0 | RP2350 / pico2 ボード |
| arm-none-eabi-gcc | xPack 14.2.1-1.1 | apt の 13.2 では Verilator runtime がリンクできず NG |
| Verilator | 5.040 | 5.048 にすると参考実装の `verilated.cpp` と ABI 不一致 |
| openocd | RasPi fork (rpi-common, SHA `acff23ff`) | RP2350 cfg 同梱 |
| `verilated.cpp` | [Soft-FPGA-TD4](https://github.com/open-tommie/Soft-FPGA-TD4) 由来の Pico パッチ版 | `firmware/src/verilib/` 配下に同梱 |

## Windows: usbipd-win

WSL2 はデフォルトで USB デバイスを見られない。`usbipd-win` で
USB を WSL2 にアタッチする。

### インストール

PowerShell（管理者）で 1 回だけ:

```powershell
winget install --interactive --exact dorssel.usbipd-win
```

インストール後に Windows を再起動。

### デバイス ID 確認

Pico 2 と Debug Probe を USB 接続した状態で:

```powershell
usbipd list
```

下記の VID:PID で識別する。

| デバイス | VID:PID | 状態 |
| --- | --- | --- |
| Pico 2 (BOOTSEL モード) | `2e8a:000f` | UF2 D&D 待ち |
| Pico 2 (アプリ起動中、stdio_usb 有効) | `2e8a:000a` | USB-CDC が `/dev/ttyACM0` |
| Pico Debug Probe (CMSIS-DAP) | `2e8a:000c` | SWD/UART ブリッジ |

### bind と attach

`bind` は 1 回だけ（管理者必要、Windows 再起動を跨いでも残る）。`attach` は WSL2 セッション
や USB 抜き差しの度に必要（管理者不要）。手動・スクリプトの両方を記す。

#### attach: 手動

PowerShell（管理者）で `usbipd list` を実行し、Pico 2 と Debug Probe の BUSID を確認:

```text
BUSID  VID:PID    DEVICE                                     STATE
2-3    2e8a:000a  USB シリアル デバイス (COM5)               Not shared
2-4    2e8a:000c  CMSIS-DAP v2 Interface                     Not shared
```

該当する BUSID を bind → attach する。BUSID は環境（USB ポート）ごとに変わる:

```powershell
# Pico 2 本体（UART モニタ等で必要なとき）
usbipd bind   --busid 2-3        # 管理者 PowerShell（1 回だけ）
usbipd attach --wsl --busid 2-3  # WSL セッションごと

# Pico Debug Probe（SWD 書き込みに必要）
usbipd bind   --busid 2-4
usbipd attach --wsl --busid 2-4
```

#### attach: スクリプト

本体は [`scripts/wsl-attach.ps1`](../scripts/wsl-attach.ps1)（PowerShell）。
`usbipd state` の JSON 出力から VID:PID で BUSID を自動検出し、未 bind / 未 attach のものだけを処理する。
`bind` 時に管理者権限が必要なら `Start-Process -Verb RunAs` で自己 UAC 昇格する。

**Windows PowerShell から実行する場合:**

```powershell
# 検出のみ
.\scripts\wsl-attach.ps1 -Mode list

# 初回: 未 bind を自動 bind（UAC ダイアログ）してから attach
.\scripts\wsl-attach.ps1 -Mode bind

# 2 回目以降: bind 済み前提で attach
.\scripts\wsl-attach.ps1
```

**WSL2 のシェルから実行する場合:**

[`scripts/wsl-attach.sh`](../scripts/wsl-attach.sh) は `powershell.exe -File wsl-attach.ps1`
を呼ぶだけの薄いラッパー。引数だけ bash 風に書ける。

```bash
scripts/wsl-attach.sh --list      # = -Mode list
scripts/wsl-attach.sh --bind      # = -Mode bind   (UAC 昇格)
scripts/wsl-attach.sh             # = -Mode attach
```

VID:PID の対応は方法 A の表のとおり（`2e8a:000f` BOOTSEL / `2e8a:000a` stdio_usb /
`2e8a:000c` Debug Probe）。スクリプトはこれを内部で持っているので、
USB ポートを差し替えて BUSID が変わっても再実行で追従する。

### detach

WSL2 から外して Windows 側の COM ポート等で使いたいとき。`attach` した瞬間
Windows 側からは見えなくなる（排他）ので、Windows 側で COM を使うときは必ず detach する。

#### detach: 手動

```powershell
usbipd detach --busid 2-3
usbipd detach --busid 2-4
```

#### detach: スクリプト

```powershell
# Windows PowerShell から
.\scripts\wsl-attach.ps1 -Mode detach
```

```bash
# WSL2 シェルから
scripts/wsl-attach.sh --detach
```

## WSL2: 確認

```bash
ls /dev/ttyACM*       # Pico を attach 済みなら /dev/ttyACM0 が見える
lsusb                 # 2e8a:000a (Pico stdio) や 2e8a:000c (Probe) が並ぶ
```

`Permission denied` が出る場合は `dialout` グループに参加:

```bash
sudo usermod -aG dialout "$USER"
# 一度ログアウト→再ログインで反映
```

## ビルドと書き込み

### Docker イメージのビルド（初回のみ、約 10 分）

```bash
USER_UID=$(id -u) USER_GID=$(id -g) \
    docker compose -f docker/compose.yml build dev
```

### ファームウェアビルド

```bash
./scripts/build-pico.sh
# → firmware/build/firmware.uf2 が生成される
```

### Hello UF2 を BOOTSEL で書き込む（Probe 不要）

1. Pico 2 の BOOTSEL ボタンを押しながら USB 接続
2. Windows のエクスプローラに `RPI-RP2350` ドライブが現れる
3. `firmware/build/firmware.uf2` をドラッグ＆ドロップ
4. 自動リセット後、オンボード LED が 2 Hz で点滅

### Debug Probe で書き込む（Probe 必須）

事前に Probe を WSL2 に attach 済みであること。

```bash
./scripts/flash.sh
# → Programming → Verified OK → shutdown
```

### USB-CDC ログを見る

WSL2 側:

```bash
sudo apt-get install -y picocom
picocom -b 115200 /dev/ttyACM0
# 終了は Ctrl-A → Ctrl-X
```

Windows 側で見るなら `usbipd detach --busid <Pico>` してから VSCode の
Serial Monitor で COM ポートを開く。

## トラブルシューティング

| 症状 | 原因 / 対処 |
| --- | --- |
| `unable to find a matching CMSIS-DAP device` | Probe を attach し忘れ。`usbipd list` で State を確認 |
| `/dev/ttyACM0` が現れない | usbipd attach が外れた、または Pico がまだ BOOTSEL のまま |
| `Permission denied` on `/dev/ttyACM0` | `dialout` グループ未参加。`usermod` 後に再ログイン |
| `usbipd attach` が "device is busy" | Windows 側で既に COM ポートを開いている。Serial Monitor を閉じる |
| Docker ビルドが jimtcl で失敗 | `--enable-internal-jimtcl` 抜けの可能性。`docker/Dockerfile` を確認 |
| firmware build が `PRIu64` / `__aeabi_read_tp` 未定義で落ちる | arm-gcc が apt の 13.2 になっている。xPack 14.2 への切替を確認 (`docker/Dockerfile` の `XPACK_ARM_GCC_VER`) |
| firmware build が `Vothello_top__Syms__Slow.cpp` not found | Verilator バージョン違いで生成ファイル名が変わった。`docker/Dockerfile` の `VERILATOR_TAG=v5.040` を確認 |
