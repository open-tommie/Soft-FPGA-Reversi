# 03-テスト戦略

実機 (Pico 2 + Debug Probe + USB-CDC) を含めた検証スタックの選定メモと初期スケルトン。

## レイヤ構成

| レイヤ | 対象 | 場所 | フレームワーク |
| --- | --- | --- | --- |
| RTL ユニット | Verilog 単体 (`legal_bb`, `flip_calc`, ...) | [`verif/tb_cocotb/`](../verif/tb_cocotb/) | cocotb (Python + pytest) |
| プロトコルシナリオ | `scenarios/*.txt` 形式の局面回帰 | [`verif/tb_protocol/`](../verif/tb_protocol/) | pytest + 自作 runner（後段） |
| HIL（実機） | Pico 2 + Debug Probe + USB-CDC | [`verif/tb_hil/`](../verif/tb_hil/) | **pytest + pyserial** |

3 つとも **pytest を runner に統一** することで `pytest verif/` 一発で回せるようにする。
リファレンス実装 [`reversi_rules.py`](../verif/golden/reversi_rules.py) を **唯一の golden** として
全レイヤから直接 import できる利点も大きい。

## HIL レイヤの選定理由

### 採用: pytest + pyserial

- 既に cocotb の関係で **pytest がツールチェインに含まれている**（追加コスト最小）
- リファレンスが Python なので `assert dut_response == reversi_rules.legal_moves(board)` がそのまま書ける
- `dut` フィクスチャを差し替えるだけで「USB-CDC 経由」と「Verilator stdin/stdout 経由」を切替可能
  → Step 2 の host 回帰と HIL でテスト本体を共用できる
- Probe 経由のフラッシュは [`scripts/flash.sh`](../scripts/flash.sh) を `subprocess` で呼ぶだけ（再実装不要）

### 検討した候補

| 候補 | 判定 |
| --- | --- |
| 素 pytest + pyserial + pexpect | 採用。最もシンプル |
| pytest-embedded (Espressif) | dut.write/expect API は便利だが ESP-IDF ファーストで Pico 用 hook 自作が必要。テスト数が増えたら検討 |
| Robot Framework + SerialLibrary | キーワード駆動で非 Python テスター向け。本プロジェクトでは過剰 |
| Unity / Ceedling | C 単体テスト向き。HIL とは目的が違う |
| Renode (ARM エミュレータ) | CI 用に強力だが Verilator 生成 C++ をリンクした実態の検証にはならない |

## tb_hil スケルトン

```text
verif/tb_hil/
  conftest.py         # フィクスチャ: flashed_firmware / dut / expect
  pytest.ini          # markers: slow, flash
  requirements.txt    # pytest, pyserial
  test_hello_uf2.py   # Step 1: "Hello, world!" の出現と 1 Hz 周期
  README.md           # 実行手順・オプション・拡張方針
```

### フィクスチャ

| 名前 | スコープ | 役割 |
| --- | --- | --- |
| `flashed_firmware` | session | `scripts/flash.sh` を **1 回だけ** 呼んで ELF を書き込む。`--no-flash` でスキップ可 |
| `dut` | function | `serial.Serial(/dev/ttyACM0, 115200, timeout=2.0)`。CDC 再列挙待ちで最大 3 秒リトライ |
| `expect` | function | 正規表現で行マッチ、タイムアウト時は直近 10 行を AssertionError に同梱 |

### CLI オプション

| フラグ | 既定 | 用途 |
| --- | --- | --- |
| `--elf PATH` | `firmware/build/firmware.elf` | 別 ELF を書き込む |
| `--port` | `/dev/ttyACM0` | USB-CDC ポート |
| `--baud` | `115200` | ボーレート |
| `--no-flash` | OFF | フラッシュをスキップ（既書き込み状態で再テスト） |

## 実行

ビルド・書き込みと同じく Docker 統一構成 (`docker/compose.yml` の `test-hil` サービス)
で動かす。コンテナ内で venv (`/opt/hil-venv`、名前付きボリューム) が作られ、
2 回目以降は再利用される。

```bash
# 実機接続準備
scripts/wsl-attach.sh           # Pico + Debug Probe を WSL2 にアタッチ
scripts/build-pico.sh           # firmware/build/firmware.elf 生成

# HIL テスト (Docker 経由)
scripts/test-hil.sh                    # 全テスト
scripts/test-hil.sh --no-flash         # 既書き込み状態で再テスト
scripts/test-hil.sh -k hello           # 名前で絞る
```

WSL2 ホスト直で動かしたい場合（依存を host 側に持つ必要あり）:

```bash
python3 -m venv .venv
.venv/bin/pip install -r verif/tb_hil/requirements.txt
.venv/bin/pytest verif/tb_hil
```

### Docker 内での docker-in-docker 回避

`flashed_firmware` フィクスチャは環境変数 `SFR_IN_CONTAINER=1`
（`compose.yml` の `test-hil` サービスで設定）を見て分岐する:

| 実行環境 | 呼ぶスクリプト |
| --- | --- |
| `test-hil` コンテナ内 | `scripts/_flash-inner.sh`（openocd を直接） |
| WSL2 ホスト直 | `scripts/flash.sh`（内部で `docker compose run flash`） |

これでコンテナ内から `docker` コマンドを呼ばずに済む。

## 拡張ロードマップ

Bootstrap Step との対応:

| Step | 追加するテストファイル | 内容 |
| --- | --- | --- |
| 1 | `test_hello_uf2.py` | `Hello, world!` の 1 Hz 周期確認（**今ここ**） |
| 2 | `test_size_baseline.py` | `arm-none-eabi-size` の出力をパースし、Verilog リンク前後の flash/SRAM 増分を assert |
| 3 | `test_proto_pi_ve.py` | `dut.write(b"PI\n")` → `expect(r"^PO ")` で UART テキストプロトコル疎通 |
| 4 | `tb_cocotb/test_legal_bb.py` | cocotb で `reversi_rules.legal_moves` と全合法局面照合（HIL ではなく RTL レイヤ） |
| 5 | `test_scenarios.py` | `verif/golden/scenarios/*.txt` をパラメトライズして PICK_FIRST 結果を検証 |
| 6 | `test_apply_move.py` | APPLY コマンドで黒/白 bitboard が更新されることを確認 |

## 注意点

- フラッシュ後の Pico 2 は新しい VID:PID (`2e8a:000a`) で再列挙される。
  `usbipd-win` の永続バインドが効いていないと `/dev/ttyACM0` が消える。
  `scripts/wsl-attach.sh` を再実行するか、`usbipd attach --auto-attach` を常駐させる
- Windows 側で Serial Monitor が同 COM ポートを開いているとアタッチが排他で失敗する。
  テスト前に閉じる
- Docker モード (`scripts/test-hil.sh`) では `/dev` をバインドしているため
  CDC 再列挙に追従できる。ホスト直モードと挙動差が出る場合は Docker モードを優先
