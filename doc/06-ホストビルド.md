# 06-ホストビルド (Pico 不要モード)

[`rtl/othello_top.v`](../rtl/othello_top.v) を Verilator で C++ 化し、
**Pico 2 実機を一切介さず Docker 内 Ubuntu 24.04 上で stdin/stdout 駆動の
単独 ELF として動かす** ためのビルド経路。

[`firmware/`](../firmware/) は Pico SDK + arm-none-eabi-gcc + 最小 Verilator
ランタイムでクロスビルドするのに対し、こちらは **ホスト gcc + 上流 Verilator
ランタイムで素直にネイティブビルド** する別ツリー。共有しているのは
`rtl/*.v` のみ。

## 何が嬉しいか

- Pico を焼かずにプロトコル骨格 (`PI`/`VE`/`ER02 unknown`) を確認できる
- `verif/golden/scenarios/*.txt` を将来
  `./reversi_host < scenario.txt` で食わせる回帰の土台になる
- `legal_bb` / `flip_calc` を Verilog に追加した時、まず host で通してから
  Pico を焼く流れに乗せられる（書き込み + ttyACM 再列挙のラグを毎回踏まない）

## 使い方

```bash
# ビルドのみ → host/build/reversi_host
scripts/build-host.sh

# ビルド後にコンテナ内で起動 (stdin から流し込む)
printf 'PI\r\nVE\r\n' | scripts/build-host.sh -- run

# 想定出力
#   PO
#   VE01reversi-fw
```

EOF を検出すると DUT に CR+LF (2 byte) を補ってから残応答を吐かせ、`tx_valid`
が一定 cycle 静まったら自動で終了する（パイプリダイレクトで綺麗に閉じる）。
入力末尾が既に CR+LF なら補わない。

## 構成

| ファイル | 役割 |
| --- | --- |
| [`host/CMakeLists.txt`](../host/CMakeLists.txt) | Pico SDK 不参照。Verilator codegen + 上流 `verilated.cpp` / `verilated_threads.cpp` をリンク |
| [`host/main.cpp`](../host/main.cpp) | `firmware/src/main.cpp` の Pico ハードウェア依存だけを `read(STDIN_FILENO)` / `fputc` に置換した版 |
| [`scripts/build-host.sh`](../scripts/build-host.sh) | ホスト wrapper (`docker compose run --rm build-host` を起動) |
| [`scripts/_build-host-inner.sh`](../scripts/_build-host-inner.sh) | コンテナ内で `cmake -GNinja` → `cmake --build` を回す inner |
| [`docker/compose.yml`](../docker/compose.yml) | `build-host` サービス定義 |

## 設計上の注意

### Verilator 5.x は `verilated_threads.cpp` も必須リンク

`/usr/local/share/verilator/include/verilated.cpp` 単独だと
`VlThreadPool::VlThreadPool(...)` が undefined reference になる。
`--threads` を渡していなくても上流 runtime が `VerilatedContext::threadPoolp()`
で参照するため、`verilated_threads.cpp` も同時に link する。
[`host/CMakeLists.txt`](../host/CMakeLists.txt) 参照。

### ホスト stdin はバックプレッシャを実装する必要がある

[`rtl/proto.v`](../rtl/proto.v) の FSM は CR+LF (`\r\n`) 受領後 `S_DISPATCH`/`S_TX`
に抜けて RX を取り込まない。Pico 版は UART (~87µs/byte) で自然に律速されるが、
ホスト版は stdin pipe から無限速で食えるので **応答中の次行が丸ごと落ちる**。

[`host/main.cpp`](../host/main.cpp) は次の簡易バックプレッシャで回避:

1. CR+LF を流したら `awaiting_response = true`
2. `tx_valid` が 1 度立ったことを記録
3. その後 `tx_valid` が連続 4 cycle (`kSettleCycles`) 0 なら
   `awaiting_response = false` に戻し、次の行の読み取りを再開

`kSettleCycles=4` は `S_DISPATCH` の 1 cycle と `S_TX` 終端の 1 cycle を
跨ぐのに十分なマージン。

### 上流 `verilated.cpp` を直接コンパイルする位置

[`firmware/CMakeLists.txt`](../firmware/CMakeLists.txt) は組込み向けに最小化した
[`firmware/src/verilator-runtime/verilated.cpp`](../firmware/src/verilator-runtime/verilated.cpp)
を使うが、ホスト版はそれを使わない（mutex / threads / OS 依存の stub を
剥がす意味がない上、上流に追従しなくなる）。`/usr/local/share/verilator/include/`
を直接ソースに加える形にしてある。

## ID コマンド（デバイス情報）

`ID\r\n` を送ると、C++ 層でインターセプトしデバイス情報を返す。

**ホスト版:**

```text
送信: ID
応答: ID pf=WSL2 Ubuntu24.04 git=abc1234 vl=5.048 bld=2026-04-29
```

**Pico 2 版:**

```text
送信: ID
応答: ID pf=rp2350-arm-s flash=4096KB prog=128KB(3%) ram=520KB bss=48KB(9%) clk=150MHz chip=B2 git=abc1234 bld=2026-04-29
```

| フィールド | 対象 | 内容 | 取得元 |
| --- | --- | --- | --- |
| `pf=` | 両方 | プラットフォーム | Pico: `PICO_PLATFORM` / ホスト: `/proc` + `/etc/os-release` |
| `flash=` | Pico | フラッシュ総量 (KB) | `PICO_FLASH_SIZE_BYTES`（コンパイル時定数） |
| `prog=` | Pico | フラッシュ使用量 KB と % | リンカシンボル `__flash_binary_end - __flash_binary_start` |
| `ram=` | Pico | SRAM 総量 (KB) | RP2350 固定値 520 KB |
| `bss=` | Pico | 静的 SRAM 使用量 KB と % | リンカシンボル `__bss_end__ - SRAM_BASE(0x20000000)` |
| `clk=` | Pico | システムクロック (MHz) | `clock_get_hz(clk_sys)` |
| `chip=` | Pico | チップリビジョン | `SYSINFO CHIP_ID[31:28]`（0=B0, 1=B1, 2=B2） |
| `git=` | 両方 | git ショートハッシュ（`+` = ダーティ） | CMake で `git rev-parse --short HEAD` |
| `vl=` | ホスト | Verilator バージョン | CMake で `verilator --version` から抽出 |
| `bld=` | 両方 | ビルド日（ISO 形式） | `__DATE__` マクロを YYYY-MM-DD に変換 |

`bss=` は data + bss セクションの末尾アドレスを表す。
ヒープ・スタックは含まない（動的使用量は省略）。

## BM コマンド（ベンチマーク）

`BM\r\n` を送ると、Verilog DUT に渡さず C++ 層でインターセプトし、
`eval()` の実測クロック情報を返す。

```text
送信: BM
応答: BM clk=3.978MHz tick=251.4ns
```

| フィールド | 意味 |
| --- | --- |
| `clk=` | 推定 soft-FPGA クロック（1 tick = clk の 1 サイクル） |
| `tick=` | `eval()` 2 回（立下り + 立上り）の実測平均時間 |

### 実装の仕組み

`host/main.cpp` と `firmware/src/main.cpp` は stdin/UART を **ラインバッファ経由**
で DUT に渡す。CR+LF が来た時点で行を検査し、`BM` なら DUT に流さず即応答する。

```text
stdin  ──▶  rx_linebuf  ──(BM?)──▶  bench_report_protocol()  → stdout
                          ──(他)──▶  DUT (1 byte/cycle でリプレイ)
```

| 環境 | 計測手段 | 精度 |
| --- | --- | --- |
| ホスト (Ubuntu) | `clock_gettime(CLOCK_MONOTONIC)` | ナノ秒 |
| Pico 2 | `time_us_32()` | マイクロ秒（tick 数が少ないと粗い） |

### ゲームプロトコルとの関係

`BM` は `proto.v` に定義のないホスト診断専用の拡張コマンドであり、
[`etc/protocol.md`](../etc/protocol.md) の正式プロトコルには含まれない。
**Pico 実機への書き込みなしに soft-FPGA クロックを実測できる**のがホスト版の主な用途。
